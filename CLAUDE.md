# CLAUDE.md — 프로젝트 운영 문서

> 이 파일은 Claude가 **매 세션 시작 시 자동으로 읽는** 문서다.
> 사용자와 Claude가 이 프로젝트를 "어떻게" 진행하는지에 대한 약속을 담는다.
> 세션에 들어오면 이 문서를 먼저 확인하고, 아래 원칙대로 작업한다.

---

## 1. 프로젝트 개요

- **무엇을**: KMP 기반 **데스크톱 AI 노트 앱**. **NVIDIA NIM 무료 API**(사용자 키)로 AI가 노트를 직접 조작(1차 텍스트 → 2차 그림). 노트는 로컬 저장.
- **어떻게**: **바이브 코딩** — 사용자는 소스코드를 직접 작성/열람하지 않는다. Claude가 기획·설계·개발·배포·수익화를 실무로 수행한다.
- **어디까지**: 기획 → 설계 → 개발 → 소스관리 → 배포 → 출시 → 유지보수 → **수익화**까지 전 과정.
- **성격**: 학습 프로젝트가 아니라 **실제 결과물**을 만드는 프로젝트.

---

## 2. 바이브 코딩 작업 원칙 (가장 중요)

사용자는 코드를 보지 않는다. 따라서 Claude는 **매 작업마다** 다음을 지킨다.

1. **무엇을 / 왜 했는지 설명** — 코드가 아니라 "결과와 의도"를 평이한 한국어로.
   - 예: "메모를 저장하면 파일이 아니라 로컬 DB에 들어가게 만들었어요. 앱을 껐다 켜도 안 사라집니다."
2. **시각화 자료 제공** — `mcp__visualize__show_widget`(위젯 렌더링 도구)로 채팅에 그림을 띄운다.
   - 구조/흐름 → 다이어그램, 화면 → 목업, 데이터 → 표/차트.
3. **작동 원리를 이해시킨다** — "이 버튼을 누르면 → 이런 순서로 동작한다"를 그림/단계로 풀어준다.
4. **전문용어는 번역해서** — 꼭 필요하면 용어를 쓰되, 바로 옆에 일상어 설명을 붙인다.
5. **매 단계 실행 검증** — 만들고 → 실제로 앱을 켜서 확인 → 사용자에게 결과(스크린샷/설명) 보고.

> 핵심 규칙: **코드 덤프 금지, 설명과 그림 우선.** 사용자가 "어떻게 작동하는지" 말로 설명할 수 있게 만드는 것이 목표다.

## 5. llm-wiki 운영 규칙 (재사용 문서 시스템)

모든 단계 산출물(기획·설계·배포 등)은 **`wiki/`** 에 보관한다. 자세한 규칙은 [wiki/README.md](wiki/README.md).

- 위키는 **재사용 껍데기(①: 폴더구조·`_templates/`·규칙) + 프로젝트 내용(②)** 로 분리. 새 프로젝트엔 ①만 복사해 재사용.
- 문서는 표준 프론트매터 + **TL;DR 3줄**로 시작 (양식: `wiki/_templates/_doc-template.md`).
- 새 문서를 만들면 `wiki/README.md`의 인덱스 라우터에 한 줄 추가.
- 사용자가 질문하면 인덱스 라우터에서 관련 문서를 찾아 TL;DR로 즉답한다.
- 진행 로그는 `wiki/_meta/changelog.md`에 누적.
- **위키는 외부 사용자도 질문 가능한 저장소**다(수동 README 대체). 리포 루트 `llms.txt`(AI용 문서 지도) + `README.md`(사람용 안내판)를 유지·갱신한다. 새 문서를 추가하면 `llms.txt`에도 반영.
- **모든 UI/화면 작업은 루트 `DESIGN.md`(Quiet Ink 디자인 시스템)를 따른다.** 색·타이포·간격·컴포넌트는 DESIGN.md 토큰으로 조립하고, 없는 값은 DESIGN.md에 먼저 추가한 뒤 쓴다. 시그니처 규칙: AI 생성물엔 보라 흔적(ai-mark)을 부착.

## 6. 소스관리 자동화

- 원격: GitHub `prgrms-my_notepad_app` (Public), 기본 브랜치 `main`. 자세히는 [wiki/04-source-control/git-strategy.md](wiki/04-source-control/git-strategy.md).
- **챕터가 마무리될 때마다 Claude가 자동으로 원격에 반영한다** — `git add -A → commit → push`. 사용자는 git 명령을 신경 쓰지 않고 학습에 집중.
- 커밋 메시지는 `type: 요약`(docs/feat/fix 등). 변경이 없으면 생략.
- 수동 반영이 필요하면: `./scripts/sync-remote.sh "메시지"`.
